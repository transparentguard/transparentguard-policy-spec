# TransparentEE — Product Specification (v1)

**Tagline:**  
“PhotoMath for Electrical Engineering that trains your brain, not just gives answers.”

**Core promise:**  
Upload any EE problem → get a source-backed, step-by-step, critical-thinking-focused solution that you can export, revisit, and learn from across all 4+ years of EE (undergrad → grad/PhD).

***

## 1. Product Overview

**Name:** TransparentEE  
**Type:** AI-native, MCP-compatible web app (PWA-friendly)  
**Target user:** Electrical Engineering students (freshman through grad/PhD)  
**Primary platform:** Web (mobile-responsive), later can wrap as PWA/native if desired  
**

Primary AI model: DeepSeek-R1 (best open model for math/EE step-by-step reasoning).
•	Secondary AI model: Llama 3.1 70B Instruct (best open model for structured explanations and introspection prompts).
•	Primary API provider: OpenRouter (unified access to both models, easy fallback routing, no markup on model pricing).
•	Backup API provider: Fireworks AI (hosts both DeepSeek and Llama with generous free/low-cost tiers, use if OpenRouter is down or rate-limited).



**Key differentiators:**

- Not just answers: **guided reasoning + introspection**.  
- **Source-backed** solutions (textbook-style references, explicit laws/theorems).  
- **Diagram/schematic-friendly** with optional user notes for context.  
- **PDF export** for every solution.  
- Designed as a **permanent EE tool** across all courses.

### 2.2 Problem upload flow

**Step 1: Upload image**

- Supported formats: JPG, PNG, HEIC, PDF (single page).  
- UI: drag-and-drop + file picker + mobile camera capture.  
- Max file size: e.g., 10 MB.  
- Auto-preview of uploaded image.



Multiple Ways to Get Help
 TransparentEE is designed to meet students wherever they are in the engineering process. While uploading a photo of a problem is the fastest option, it isn’t the only one. Students can also type a question or describe a problem in their own words, or use voice input to explain what they’re working on hands-free. This is especially useful during labs, hardware projects, breadboarding, debugging, PCB work, robotics, embedded systems, or any situation where there isn’t a clean diagram or textbook problem to photograph. Whether the challenge starts with an image, a written description, or a spoken explanation, TransparentEE analyzes the context and provides the same structured, guided learning experience focused on understanding—not just answers.



**Step 2: Add context (optional but encouraged)**

- Text area: “Add any extra notes to help the AI (assumptions, simplifications, what you’re stuck on).”  
- Example prompts in placeholder:
  - “Assume ideal op-amp.”  
  - “Steady-state AC, 60 Hz.”  
  - “Ignore R3 for now.”  
  - “I don’t understand how they got the transfer function.”

**Step 3: Select focus mode (optional)**

- Default: “Full solution + reasoning.”  
- Optional toggles:
  - “More hints, fewer steps shown at once” (guided mode).  
  - “Show all steps at once” (reference mode).  

**Step 4: Submit**

- Button: “Solve with TransparentEE”  
- Loading state: shows “Analyzing problem… Identifying topic… Building solution…”

### 2.3 Solution generation & presentation

**Backend process (high level):**

1. Image preprocessing (enhance contrast, deskew if needed).  
2. OCR + layout detection (text + equations; basic diagram/schematic parsing where possible).  
3. Topic classification:
   - Circuits, Signals & Systems, Electronics, Control, Power, EM, RF, Comms, DSP, etc.  
4. Problem type inference:
   - E.g., “nodal analysis,” “transfer function derivation,” “fault current calculation,” “Bode plot sketch,” etc.  
5. Solution generation via AI:
   - Step-by-step reasoning.  
   - Alternative methods (if applicable).  
   - Source references (book + chapter/example-level).  
   - Introspection prompts.  
6. Post-processing:
   - Format math nicely (LaTeX-style rendering).  
   - Structure into sections.  
   - Attach metadata (topic, course level, tags).

**Frontend solution view:**

Organized into clear sections:

1. **Problem summary**
   - Restated problem in text (as understood by AI).  
   - List of knowns/unknowns.  
   - Any assumptions (from image + user notes).

2. **Approach overview**
   - High-level method: “We’ll use nodal analysis with KCL at nodes A and B.”  
   - Why this method: 1–2 sentences.

3. **Guided steps** (core of TransparentEE)

   Default: **guided mode** (encourages critical thinking):

   - Steps are shown one at a time or in small groups.  
   - Each step includes:
     - What to do next.  
     - Why (law/theorem/principle).  
     - Sometimes a short question/prompt:
       - “Which node would you choose as reference here, and why?”  
       - “Before calculating, what form do you expect the transfer function to take?”  
   - User clicks “Show next step” to continue.

   Option: **reference mode**:
   - All steps shown at once, like a traditional solution.

   Each step includes:
   - Math/equations (rendered nicely).  
   - Short explanation.  
   - citations tag (e.g., “[Nilsson & Riedel, Ch. 4]”).

4. **Alternative methods** (if applicable)

   - Section: “Other ways to solve this”  
   - For each method:
     - Brief description.  
     - When it’s preferable.  
     - Pros/cons vs the primary method.

5. **Source-backed reasoning**

   - Section: “Why this works (sources)”  
   - Lists:
     - Key laws/theorems used (KVL, KCL, superposition, final value theorem, etc.).  
     - Canonical references:
       - “Similar to Example 4.3 in Nilsson & Riedel, Electric Circuits.”  
       - “See Ogata, Modern Control Engineering, Ch. 5 (root locus design).”  
   - Short explanation of how the problem maps to those references.

Part 2: Backend Implementation Spec (The Knowledge Engine)
You are building a Keyword-Routed Retrieval Engine. Because you are avoiding embedding models, you will use a direct file-lookup system to inject knowledge. This is fast, cheap, and highly predictable.
1. The Directory Structure (The Backend "Brain")
Organize your backend storage to allow the AI to "navigate" these books like a library. Create a directory called /knowledge_base.
/backend/knowledge_base/
├── circuits/
│   ├── foundations_analog_digital.txt
│   └── lessons_circuits_semiconductors.txt
├── signals_systems/
│   └── signals_and_systems.txt
├── electronics/
│   └── semiconductor_devices_fiore.txt
├── electromagnetics/
│   └── em_vol1_ellingson.txt
├── logic/
│   └── digital_logic_brown.txt
├── control/
│   └── control_systems_oer.txt
├── power/
│   └── power_system_analysis_oer.txt
└── communications/
    └── communication_systems_oer.txt

2. The Logic (The "Router")
Your backend code (Node/Python) will act as the librarian. When a user uploads a problem:
Classification Step: Your classification prompt (which runs before the solution generation) identifies the topic.
Lookup Step: The backend maps that topic to one of the files above.
Extraction Step: The system performs a "Text Chunk" retrieval (e.g., pulling the first 3000 tokens of the relevant section).
3. The "Baked-In" Injection (The System Prompt)
This is the final step. You don't "train" the model; you feed it the reference material in the prompt context. Your API call payload to DeepSeek should look like this:
{
  "model": "deepseek-r1",
  "messages": [
    {
      "role": "system",
      "content": "You are a professional EE research assistant. Use the following context as the source of truth for definitions and methods: [INJECTED TEXT FROM RELEVANT TXT FILE]. Cite your sources clearly."
    },
    {
      "role": "user",
      "content": "[STUDENT PROBLEM + CONTEXT NOTES]"
    }
  ]
}

Why this works:
Zero Latency: You are simply reading a text file from your own server and passing it to the AI.
Zero Training Cost: You are using the AI's "Reasoning" capability to apply existing knowledge, not trying to burn compute to teach it new facts.
High Accuracy: By placing the source text in the system message, you effectively tell the AI, "Don't guess; use this book to figure it out."
This plan allows you to launch with full research-grade capabilities on day one, and as your library grows, you simply add more .txt files to the relevant folders.

## 3. Technical Architecture (High Level)

### 3.1 Frontend

- Stack: React/Next.js or similar (you can build on Replit, then deploy).  
- Features:
  - Auth (email/password + OAuth).  
  - Image upload UI.  
  - Context notes input.  
  - Solution viewer (guided + reference modes).  
  - Library/archive UI.  
  - PDF export (client-side or server-side generation).  
- Responsive design: mobile-first (students will use phones in class/study).

### 3.2 Backend

- Stack: Node/Express or Python/FastAPI (your choice).  
- Responsibilities:
  - Auth & session management.  
  - File upload handling (store images in S3-compatible bucket).  
  - Orchestration of AI pipeline.  
  - Solution storage (DB).  
  - PDF generation endpoint.  
  - Subscription management (Stripe integration).

### 3.3 AI pipeline

- Image preprocessing (OpenCV or similar, optional in v1).  
- OCR + equation detection:
  - Use existing OCR services or open-source models.  
- Topic & problem-type classification:
  - Fine-tuned or prompted LLM with EE knowledge.  
- Solution generation:
  - LLM with:
    - System prompt emphasizing:
      - Step-by-step reasoning.  
      - Source-backed explanations.  
      - Introspection prompts.  
      - Alternative methods.  
    - BYOM support:
      - User can configure API key/model in settings.  
      - Default model provided for those who don’t.  
- Post-processing:
  - Structure output into JSON sections (summary, steps, sources, prompts, etc.).  
  - Render math (LaTeX to MathML/HTML via library).

### 3.4 Database

- Tables/collections:

1. `users`
   - id, email, password_hash, created_at, plan_type, stripe_customer_id, settings (BYOM config, etc.).

2. `problems`
   - id, user_id, image_url, context_notes, focus_mode, topic, course_level, created_at.

3. `solutions`
   - id, problem_id, structured_content (JSON), created_at.

4. `user_notes`
   - id, problem_id, note_text, created_at (for introspection answers).

5. `subscriptions`
   - id, user_id, plan_type, start_date, end_date (if applicable), status.
